# 設計ルール

## YAGNI: 今必要なものだけ作る

- 「将来必要になるかもしれない」コードは書かない
- 未使用のメソッド、フィールド、パラメータは即座に削除
- 1箇所でしか使わないヘルパー関数や抽象化は作らない
- Feature flagや後方互換のためのshimは不要。直接コードを変更する

## 凝集度と結合度

- 1つのモジュール/関数の変更理由は1つだけにする
- 説明が「AとBとCをする」になったら分割を検討する
- モジュール間の依存は抽象（インターフェース/型）を通す
- 外部依存（API、DB、ファイルシステム）は境界で抽象化し、テスト可能にする

## 命名

曖昧な動詞を避け、具体的なアクションを表す名前をつける:

- `check` → `validate`, `compare`, `exists`
- `process` → `parse`, `transform`, `aggregate`
- `handle` → `route`, `dispatch`, `recover`
- `do` → 具体的な動詞に置き換える

戻り値の型名も操作内容を反映させる（`Result` → `ValidationResult`, `ParsedConfig`）

## コロケーション

関連するファイルは物理的に近くに配置する:

- テストはソースと同じディレクトリに置く
- 機能単位でディレクトリを分ける

## エラーハンドリング

- エラーは握りつぶさない。処理するか、呼び出し元に伝播する
- エラーメッセージには「何が起きたか」と「何を期待していたか」を含める
- 内部コードのバリデーションは不要。システム境界（ユーザー入力、外部API）でのみ検証する
