# Ruby 設計ルール

## クラス設計

- 1クラス1責務。クラスの説明に「と」が含まれたら分割する
- 継承よりコンポジションを優先する
- Moduleのmixinは共通の振る舞いの共有に限定する。状態を持たせない
- `Struct` や `Data`（Ruby 3.2+）で値オブジェクトを表現する

## Rails プロジェクト

- Fat Modelを避ける。ビジネスロジックはService Object / POROに分離する
- Concernは共通のDBスコープやコールバックに限定する。ロジックを入れない
- コントローラは薄く保つ。パラメータの受け取りとレスポンスの返却のみ
- N+1クエリに注意する。`includes` / `preload` で事前読み込みする

## 依存関係

- 外部サービスへのアクセスはクラスで抽象化し、依存性注入する
- gemの選定は活発にメンテされているものを選ぶ
- gem追加前に標準ライブラリで解決できないか検討する

## ファイル構成

- 1ファイル1クラス/1モジュールを基本とする
- ネームスペースはディレクトリ構造と一致させる
- `lib/` 配下はautoloadの規約に従う
