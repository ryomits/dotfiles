# テスト共通ルール

## 基本原則

### テストは仕様である

テストは実行可能なドキュメント。システムが何をするかを明確に伝える。

### 実装ではなく振る舞いをテストする

外部から観測可能な振る舞いをテストする。実装の詳細はテストしない。

**リトマス試験**: テストが失敗した時、ユーザーにとって何が壊れたか説明できるか？できないなら振る舞いのテストではない。

```
// Bad: 設定値のアサーション（トートロジー）
assert(config.enabled == true)

// Good: 振る舞いのテスト
// 「ドキュメントが開かれた時、サーバーはXXを行う」
```

**リファクタリングでテストが壊れるなら、実装の詳細をテストしている。**

### テストの独立性

- 実行順序に依存しない
- 可変状態を共有しない
- 作成したリソースをクリーンアップする

## テスト構造

### AAA (Arrange-Act-Assert) パターン

```javascript
test('should calculate total with tax', () => {
  // Arrange
  const cart = new ShoppingCart()
  cart.addItem({ price: 100, quantity: 2 })

  // Act
  const total = cart.calculateTotal(0.1)

  // Assert
  expect(total).toBe(220)
})
```

## 命名規則

シナリオと期待される振る舞いを記述する:

```
// パターン: 関数名_条件_期待結果
test_parse_version_with_invalid_input_returns_error
test_calculate_total_with_empty_cart_returns_zero
```

## アサーション

### 文字列は完全一致を使う

```
// Good: 意図しないメッセージ変更を検出できる
assert(message == "Update available: 3.0.0 -> 4.0.0")

// Bad: メッセージが根本的に間違っていてもパスする
assert(message.contains("3.0.0"))
```

### 構造体は全体を比較する

```
// Good: 欠落や不正なフィールドを検出できる
assert(result == expected_struct)

// Bad: 新規追加や変更されたフィールドを見逃す
assert(result.name == "lodash")
assert(result.version == "4.17.21")
```

## モック

実際の実装を優先する。モックは以下の場合のみ:

- 外部ネットワーク呼び出し
- 分離が必要なファイルシステム操作
- 時間依存の振る舞い
- 非決定的な操作

## パラメータ化テスト

TDDのREFACTORフェーズで検討する。

**変換する場合:**
- テスト構造が同一で、入力値と期待出力のみが異なる

**変換しない場合:**
- セットアップがケース間で大きく異なる
- 各テストが異なる振る舞いを検証している

## セルフレビューチェックリスト

テスト実装後、各テストを確認する:

1. **振る舞いのテストか？** — 失敗時にユーザーへの影響を説明できるか
2. **トートロジーではないか？** — getterがsetした値を返すだけのテストになっていないか
3. **冗長なヘルパーはないか？** — 1行で済むものをヘルパー関数に抽出していないか
4. **既存パターンを盲目的に踏襲していないか？** — 各テストを個別にルールに照らして評価する

いずれかに該当したら、先に進む前にテストを修正する。
