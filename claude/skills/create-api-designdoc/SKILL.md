# API Design Doc 作成スキル

REST APIエンドポイントのDesign Docを対話的に作成する。テンプレートに沿ってユーザーと対話しながら各セクションを埋めていく。

## ワークフロー

### ステップ1: 基本情報の収集

AskUserQuestionで以下を確認する:

1. **HTTPメソッドとパス**: エンドポイントの仕様（例: `GET /api/v1/users/myself/session_state`）
2. **エンドポイントの目的**: 何をするAPIか
3. **出力先**: ファイルパス（デフォルト: `/tmp/` 配下）

### ステップ2: リクエスト仕様の収集

対話的に以下を確認する:

1. **認証の有無**: 認証必須か、不要か
2. **パスパラメータ**: あれば定義
3. **クエリパラメータ**: あれば定義
4. **リクエストボディ**: あれば定義と具体例

不要な項目はスキップする。

### ステップ3: レスポンス仕様の収集

対話的に以下を確認する:

1. **成功時のレスポンスボディ**: フィールド定義と具体例
2. **エラーレスポンス**: 返却しうるエラー一覧

### ステップ4: 内部処理の設計

対話的に以下を確認する:

1. **シーケンス図**: 処理の流れをmermaid sequenceDiagramで作成。参加者（API Client、サーバー、DB等）を確認する
2. **処理詳細**: シーケンス図の各ステップの補足

### ステップ5: Design Doc の生成

収集した情報を[テンプレート](template.md)に沿って出力する。

### ステップ6: レビューと修正

生成したDesign Docをユーザーに提示し、修正があれば対話的に反映する。
